# 多账号与负载均衡改造路线图

## 背景

当前网关是**单账号架构**：启动时只初始化一个 `KiroAuthManager`、一个共享 `httpx.AsyncClient`、一个全局请求限流器、一个全局模型缓存与解析器，然后 OpenAI / Anthropic 两条路由都直接从 `app.state` 读取这些单例来执行请求。

这意味着：

- 吞吐上限受单账号约束
- 某个账号异常会直接影响全部请求
- `/health`、`/v1/models`、模型解析与重试逻辑都是单账号视角

这次改造的目标不是一次性堆满所有能力，而是**按优先级分阶段落地**：

- 模式：**共享账号池**
- 路由粒度：**请求级选路**
- 配置方式：**本地配置文件**
- 优先级：**提高并发 > 提高稳定性 > 精细治理**

---

## 总体策略

推荐路线：

1. 先做**最小可用多账号池**，尽快把吞吐提起来
2. 再做**池级可观测性**，让系统变得可排查
3. 再做**故障摘除与恢复**，提升稳定性
4. 最后再做**资源治理与高级策略**

核心原则：

- **复用现有单账号逻辑**，不要重写 token 刷新、请求重试、converter、streaming formatter
- 多账号改造主要集中在：
  - **运行时编排层**
  - **路由选路层**
  - **池级状态管理层**
- 流式请求一旦选定账号，**整个生命周期固定该账号**，不做中途切换

---

## Phase 1 — 最小可用版本（最高优先级）

### 目标

让网关具备“多个账号可配置、每个请求自动选择一个账号执行”的能力，在不大改现有请求链路的前提下，先把并发吞吐抬起来。

### 要做的事

#### 1. 引入本地多账号配置文件

不要继续把多账号硬塞进 `.env`，改为单独的本地配置文件，支持账号列表结构。

每个账号至少应包含：

- `id`
- `enabled`
- `weight`
- `region`
- `profile_arn`
- `creds_file` / `sqlite_db` / `refresh_token`

#### 2. 新增账号池运行时抽象

新增账号池相关对象：

- `AccountConfig`：单个账号的静态配置
- `AccountRuntime`：单个账号的运行态
  - `KiroAuthManager`
  - `ModelInfoCache`
  - `ModelResolver`
- `AccountPool`：负责管理全部账号、启动初始化、挑选可用账号
- `SelectionResult`：一次请求的最终选路结果

#### 3. 在路由层加入请求级选路

在以下位置接入账号池：

- `kiro/routes_openai.py`
- `kiro/routes_anthropic.py`

链路建议为：

1. 收到请求
2. 从账号池拿候选账号
3. 按请求模型过滤候选账号
4. 使用加权轮询挑选一个账号
5. 用该账号自己的 `auth_manager` / `model_resolver` 执行整次请求

#### 4. 保持单账号内重试逻辑不变

继续复用当前 `KiroHttpClient` 的单账号重试逻辑：

- 403 refresh
- 429 backoff
- 5xx backoff
- timeout retry

本阶段**不做跨账号重试**。

### 本阶段不做什么

- 不做会话粘性
- 不做后台管理界面
- 不做自动摘除
- 不做账号级限流
- 不做复杂调度算法

### 为什么 Phase 1 先做

这是最小闭环：

- 配置多个账号
- 启动时初始化账号池
- 请求可以自动选账号

只要这三件事跑通，吞吐和可用性就已经会有明显提升，而且对现有 converter / streaming 链路侵入最小。

### 主要改动文件

- `main.py`
- `kiro/config.py`
- `kiro/routes_openai.py`
- `kiro/routes_anthropic.py`
- `kiro/cache.py`
- `kiro/model_resolver.py`
- `kiro/auth.py`
- 新增账号池模块（建议 `kiro/account_pool.py`、`kiro/account_types.py`）

### 验收标准

- 能配置 2 个以上账号
- 网关能启动，且初始化多个账号成功
- OpenAI / Anthropic 的流式与非流式请求都能跑通
- 日志能看到每个请求最终选中了哪个账号

---

## Phase 2 — 池级可观测性与兼容能力（第二优先级）

### 目标

让系统在多账号模式下“能看、能解释、能排查”，补齐 `/health` 与 `/v1/models` 的池化语义。

### 要做的事

#### 1. 重构 `/health`

从单账号快照升级为账号池快照：

- 保留现有全局字段
- 新增池级摘要：
  - 总账号数
  - 可用账号数
  - 调度策略
  - 池总请求情况
- 新增账号级列表：
  - auth 类型
  - region
  - token 过期状态
  - refresh 失败次数
  - 模型缓存情况
  - 当前是否可选

#### 2. 重构 `/v1/models`

从“单账号模型列表”变成“账号池模型并集”：

- 模型来源于全部可用账号的并集
- 保留 alias / hidden model 行为
- 某模型即使只在部分账号可用，也照常出现在 `/v1/models`

#### 3. 补充选路日志与初始化日志

日志建议至少包含：

- 哪些账号启动成功 / 失败
- 某次请求按模型筛选后剩下哪些候选账号
- 最终命中了哪个账号

### 本阶段不做什么

- 不做自动摘除
- 不做账号级并发上限
- 不做复杂恢复逻辑

### 为什么放在第二阶段

Phase 1 解决“能跑”；Phase 2 解决“跑起来后能看懂”。

没有池级健康状态和模型并集视图，后续做稳定性优化和调度策略很容易变成黑盒调参。

### 主要改动文件

- `main.py`
- `kiro/routes_openai.py`
- `kiro/model_resolver.py`
- `kiro/cache.py`
- 账号池模块

### 验收标准

- `/health` 能返回池级视图
- `/v1/models` 能返回模型并集
- 日志能清晰看出账号初始化、筛选、选路结果

---

## Phase 3 — 稳定性增强：账号摘除与自动恢复（第三优先级）

### 目标

当某个账号出现连续 429、5xx、refresh 失败或断流时，网关可以自动绕开这个坏账号，避免拖累整个池子。

### 要做的事

#### 1. 建立账号失败计数

为每个账号统计：

- 429 次数
- 持续 5xx 次数
- token refresh 失败次数
- 首 token 前断流次数

#### 2. 临时摘除不可用账号

达到阈值后：

- 将账号标记为不可选
- 设置摘除窗口（例如 1 分钟 / 5 分钟）

#### 3. 到期自动恢复

恢复策略第一版建议很简单：

- 到时间自动恢复候选资格
- 暂时不做主动探测

#### 4. 将执行结果回灌给账号池

当前错误分类逻辑主要在路由与 `KiroHttpClient` 内部，多账号后要把关键失败结果回传给账号池，作为摘除依据。

### 本阶段不做什么

- 不做流式请求跨账号透明重放
- 不做成功率/延迟评分
- 不做智能调度算法

### 为什么放在第三阶段

账号摘除依赖前两阶段已经完成：

- 已有账号池运行时
- 已有请求级选路
- 已有池级观测

否则一上来做摘除，错误处理链路会很乱。

### 主要改动文件

- `kiro/http_client.py`
- `kiro/routes_openai.py`
- `kiro/routes_anthropic.py`
- 账号池模块
- `/health` 输出相关代码

### 验收标准

- 人为打坏一个账号后，请求能绕开它
- 被摘除账号不会继续被选中
- 到恢复窗口后账号重新进入候选池
- `/health` 可观察到摘除状态

---

## Phase 4 — 资源治理：账号级限流与细粒度策略（第四优先级）

### 目标

避免虽然有多个账号，但热点还是集中打到某个账号，导致局部瓶颈和限流。

### 要做的事

#### 1. 增加账号级并发限制

保留现有全局 semaphore，同时为每个账号增加独立并发上限。

调度时：

- 先看账号是否支持模型
- 再看账号是否已达到并发上限
- 超限账号跳过

#### 2. 细化账号权重

支持主账号、备用账号不同权重：

- 常规时主账号承担更多流量
- 备用账号平时低权重，但仍保持可用

#### 3. 支持模型级范围控制

账号可声明：

- `allowed_models`
- `blocked_models`

便于未来做：

- 某些模型只走部分账号
- 不同模型进入不同账号子池

### 为什么放在第四阶段

这是资源治理问题，不是系统能否工作的第一优先级。应该在多账号基本稳定后再做。

### 主要改动文件

- `kiro/request_limiter.py`
- `main.py`
- `kiro/routes_openai.py`
- `kiro/routes_anthropic.py`
- `kiro/config.py`
- 账号池模块

### 验收标准

- 高并发下单账号不会被打爆
- 权重配置生效
- 模型级过滤规则生效

---

## Phase 5 — 高级能力（第五优先级）

### 目标

给后续更复杂的场景留升级空间，但不建议第一版就做。

### 可选能力

#### 1. 会话粘性 / 客户端绑定

例如：

- 同一类请求尽量固定账号
- 某个 API Key 绑定特定账号组

#### 2. 跨账号重试

适用于非流式请求：

- 一个账号失败后，切到另一个账号重试

#### 3. 外部账号管理器

将本地配置文件进一步升级为：

- 独立管理入口
- 动态加载账号
- 更细的运营治理能力

#### 4. 调度策略升级

例如：

- 最少连接
- 成功率权重
- 延迟评分
- 热点保护

### 为什么最后做

这些能力价值高，但复杂度也高，特别会影响：

- 调试难度
- 请求幂等性判断
- 流式请求一致性
- 代码维护复杂度

所以不应该在前几阶段混进主链路。

---

## 状态隔离边界

### 从 Phase 1 起必须按账号隔离

- `KiroAuthManager`
- token refresh lock
- `ModelInfoCache`
- `ModelResolver`
- token refresh task

### 可以先保持全局

- `FastAPI` app
- 调试日志中间件
- 全局请求计数
- truncation recovery 状态
- 全局 semaphore

### 到 Phase 4 再细化

- 账号级 semaphore
- 模型子池路由规则

---

## 现有能力优先复用

优先复用这些现成逻辑：

- token 生命周期与刷新：`kiro/auth.py`
- 凭证读写：`kiro/auth_storage.py`
- 单账号请求重试：`kiro/http_client.py`
- 路由共享执行链：`kiro/request_executor.py`
- 模型名规范化与 passthrough：`kiro/model_resolver.py`
- 当前全局并发限制：`kiro/request_limiter.py`

核心原则：

> 多账号改造应主要发生在“运行时编排层”和“路由选路层”，不要把 converter、streaming formatter、tokenizer 这些与账号池无关的模块一起卷进去。

---

## 推荐落地顺序

1. **Phase 1**：多账号配置 + 账号池 + 请求级选路
2. **Phase 2**：`/health` 与 `/v1/models` 的池化语义
3. **Phase 3**：账号摘除 / 自动恢复
4. **Phase 4**：账号级限流 / 权重 / 模型子池
5. **Phase 5**：会话粘性 / 跨账号重试 / 外部管理器

---

## 每阶段验证建议

### Phase 1

- 配置 2 个以上账号
- 启动成功
- OpenAI / Anthropic 各跑 1 次流式与非流式请求
- 日志能看到选中的账号

### Phase 2

- `/health` 返回账号池视图
- `/v1/models` 返回模型并集
- 日志能解释候选筛选过程

### Phase 3

- 人为让某个账号失效
- 请求自动绕开该账号
- 恢复窗口后账号重新进入池

### Phase 4

- 高并发压测下，单账号不会被打爆
- 权重与限流规则按预期生效
